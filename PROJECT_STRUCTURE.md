# Project Structure

One Next.js 15 App Router project at the repository root, plus one
sibling folder for the (separately-deployed, non-Next.js) booking
backend. No duplicate trees, no nested projects.

## Why `apps-script/` lives here but isn't "part of the project"

`Code.gs` and `Config.gs` are Google Apps Script source — they don't run
in Next.js, aren't bundled by Vercel, and aren't written in a language
`tsc`/webpack touch. They're kept in this repository purely so the
booking backend's source is version-controlled alongside the frontend
that calls it, and so that context isn't lost. Deploying them is a
separate, manual step in the Google Apps Script editor (see
`DEPLOYMENT.md`) — Vercel's build simply ignores this folder, since
Next.js only looks at `app/`, `public/`, and whatever those import.

## Conventions used throughout `components/`

- **Section folders** (`hero/`, `capabilities/`, `process/`, `solution/`,
  `cta-footer/`) each hold everything specific to that one section: its
  sub-components and its `*Data.ts` content/config file(s).
  `components/sections/*.tsx` is the thin top-level component for each
  `<section>` that composes those pieces — this is what `app/page.tsx`
  actually imports.
- **`components/booking/`** and **`components/about/`** are global
  (not page-section-scoped) — both are modals mounted once, at the root
  layout, and opened from anywhere via a React Context + hook
  (`useBookingModal()`, `useAboutModal()`).
- **`components/ui/`** holds only truly generic, presentation-only
  primitives with no feature-specific knowledge — safe to import from
  any section.
- Every section's animation variants live in
  `animations/<section>Animations.ts`, imported only by that section's
  own components — kept separate (rather than one shared file) so each
  section's motion language can be tuned independently without risking
  cross-section regressions.

## One project root, confirmed

There is exactly one `package.json`, one `tsconfig.json`, one
`next.config.js`, one `app/` directory, and one `node_modules/` (created
by `npm install`, gitignored, not included in source delivery). No
nested Next.js project, no duplicate component trees, no orphaned backup
folders — all previously present duplicates were removed as part of this
cleanup (see `CHANGELOG.md`).
