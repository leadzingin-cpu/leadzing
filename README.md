# LeadZing — Google Sheets Booking Backend

Everything needed to make the "Book Discovery Call" popup actually save
submissions to a Google Sheet. No frontend design, layout, or styling was
touched — only the functional wiring needed to send data somewhere real.

## What's in this package

```
apps-script/
  Code.gs                                  # The backend — entry points + all logic
  Config.gs                                # Sheet name, required fields, column order

frontend/leadzing-hero/
  lib/bookingApi.ts                        # NEW — fetch wrapper the modal calls
  .env.local.example                       # NEW — where the deployment URL goes
  components/booking/BookingModal.tsx      # MODIFIED — calls submitBooking() on confirm
  components/booking/ConfirmStep.tsx       # MODIFIED — adds a loading spinner + error text to the existing button

README.md                                  # This file
```

Only four frontend files are touched, and three of those four are brand
new. `BookingModal.tsx` and `ConfirmStep.tsx` keep every existing class
name, animation, and layout exactly as it was — the only additions are a
`disabled` state on the two buttons while a request is in flight, a small
spinner inside the existing "Confirm Booking" button label, and an error
line in the same style already used elsewhere in the form (`DetailsStep.tsx`'s
validation message). Nothing about the previously-approved Confirm step
layout fix is touched.

## 1. Sheet structure

**Spreadsheet name:** `LeadZing Leads`
**Tab name:** `Leads` (created automatically the first time a booking comes in, or by running `setupSheet` manually — see step 4)

| Booking Date | Booking Time | Full Name | Business Name | Email | Phone | Website / Instagram | Purpose | Goals | Submitted At |
|---|---|---|---|---|---|---|---|---|---|
| 2026-07-18 | 8:00 PM | Farhat Khan | | owwzraza@gmail.com | 09874743024 | | Other | | 2026-07-16 17:52:03 |

- The first nine columns come directly from the booking form.
- **Submitted At** is generated by the server (Apps Script), not the browser — never trust a client-supplied timestamp for a record like this.
- Optional fields (Business Name, Website / Instagram, Goals) are stored as empty strings rather than left `undefined`, so every row has the same shape.

## 2. Deployment guide

### Step 1 — Create the Google Sheet
1. Go to [sheets.google.com](https://sheets.google.com) → **Blank spreadsheet**.
2. Rename it (top-left) to exactly **LeadZing Leads**.
3. You can leave the default tab as-is — the script creates its own **Leads** tab automatically.

### Step 2 — Open Apps Script
1. In the sheet, go to **Extensions → Apps Script**.
2. This opens a script bound to this specific spreadsheet — that's what lets `Code.gs` find it automatically with zero configuration.

### Step 3 — Paste the code
1. Delete the placeholder `function myFunction() {}` in the default `Code.gs` file.
2. Paste in the contents of this package's `apps-script/Code.gs`.
3. Click the **+** next to "Files" → **Script** → name it `Config` → paste in `apps-script/Config.gs`'s contents. (The file name doesn't need the `.gs` extension typed in — Apps Script adds it.)
4. Save (Ctrl/Cmd+S), or the disk icon.

### Step 4 — (Optional) Pre-create the header row
1. In the function dropdown at the top of the editor, select **setupSheet**.
2. Click **Run**.
3. The first time you do this, Google will ask you to authorize the script (it needs permission to edit the spreadsheet) — click through **Review permissions → (your account) → Advanced → Go to (project name) → Allow**.
4. Check the sheet — a **Leads** tab now exists with a bold header row.

This step is optional — `doPost` creates the tab and headers automatically the first time a real booking comes in — but running it once lets you confirm authorization works before you're relying on a live booking to surface any permission issues.

### Step 5 — Deploy as a Web App
1. Top-right, click **Deploy → New deployment**.
2. Click the gear icon next to "Select type" → **Web app**.
3. Fill in:
   - **Description**: `LeadZing booking endpoint` (anything you like)
   - **Execute as**: **Me** (your account) — this is what lets the script write to the sheet regardless of who submits the form
   - **Who has access**: **Anyone** — this is what lets your live website (a different origin) call it. "Anyone" here means anyone can *call the endpoint*, not that they can see your sheet; the sheet itself is never made public by this setting.
4. Click **Deploy**.
5. Authorize again if prompted (same flow as Step 4).

### Step 6 — Copy the deployment URL
1. After deploying, a dialog shows a **Web app URL** — it looks like:
   `https://script.google.com/macros/s/AKfycb.../exec`
2. Copy it. This is the only value the frontend needs.

> **Redeploying later?** If you edit `Code.gs` after this, existing integrations keep working, but to publish the change you need **Deploy → Manage deployments → (pencil icon) → New version → Deploy**. Editing the code alone does not update the live `/exec` URL's behavior until you do this.

### Step 7 — Add the URL to the Next.js project
1. In your project root, copy `.env.local.example` to `.env.local` (if `.env.local` already exists, just add the line below to it).
2. Set:
   ```
   NEXT_PUBLIC_BOOKING_API_URL=https://script.google.com/macros/s/AKfycb.../exec
   ```
   (your real URL from Step 6)
3. Restart `npm run dev` (env vars are only read at server start).
4. For production (Vercel, etc.), add the same variable in your host's environment variable settings and redeploy.

### Step 8 — Test the booking flow
See the Testing section below.

## 3. Frontend integration (how it's wired)

`lib/bookingApi.ts` exports one function:

```ts
await submitBooking({
  bookingDate: "2026-07-18",
  bookingTime: "8:00 PM",
  fullName: "Farhat Khan",
  businessName: "",
  email: "owwzraza@gmail.com",
  phone: "09874743024",
  website: "",
  purpose: "Other",
  goals: "",
});
```

`BookingModal.tsx`'s `handleConfirm` calls this when the user clicks
**Confirm Booking**, and only moves to the success screen if it resolves
without throwing:

```ts
const handleConfirm = async () => {
  if (!selectedDate || !selectedTime || isSubmitting) return;
  setSubmitError(null);
  setIsSubmitting(true);
  try {
    await submitBooking({ /* mapped from wizard state */ });
    goTo("success", 1);
  } catch (err) {
    setSubmitError(err instanceof Error ? err.message : "Something went wrong. Please try again.");
  } finally {
    setIsSubmitting(false);
  }
};
```

**Loading state:** `isSubmitting` disables both the "Back" link and the
"Confirm Booking" button and swaps the button's label for a small spinner
+ "Booking..." — reusing the existing `Button` component's `disabled`
styling, no new visual language introduced.

**Error handling:** any thrown error (network failure, non-200 response,
malformed JSON, or `{ success: false, error: "..." }` from the backend)
is caught and shown as a short red line under the button, in the same
typographic style already used for validation errors in `DetailsStep.tsx`.
The user's entered data is preserved — they can just click the button
again.

### Why `Content-Type: text/plain` instead of `application/json`

Google Apps Script Web Apps don't support the CORS preflight (`OPTIONS`)
request that a `fetch()` with `Content-Type: application/json` triggers.
Sending `text/plain` keeps the browser's request a CORS "simple request"
(no preflight), while the body is still valid JSON — `Code.gs` parses it
directly from the raw request text regardless of the declared content
type. This is documented in a comment directly above the `fetch()` call
in `bookingApi.ts` so it doesn't get "fixed" back to `application/json`
by a future edit.

## 4. Testing instructions

### A. Test the Apps Script directly (no frontend needed)
1. In the Apps Script editor, open the deployment URL from Step 6 in a new browser tab.
2. You should see: `{"success":true,"message":"LeadZing booking endpoint is live. Submit bookings via POST."}` — confirms the deployment itself is reachable.

### B. Test a submission with curl (or Postman)
```bash
curl -X POST "https://script.google.com/macros/s/AKfycb.../exec" \
  -H "Content-Type: text/plain;charset=utf-8" \
  -d '{"bookingDate":"2026-07-18","bookingTime":"8:00 PM","fullName":"Test User","businessName":"Test Co","email":"test@example.com","phone":"9999999999","website":"","purpose":"Other","goals":"Just testing"}'
```
Expect: `{"success":true,"message":"Booking received. We will confirm your meeting shortly."}`
Then check the **Leads** tab — a new row should appear within a few seconds.

### C. Test validation
Send the same request with `"email":"not-an-email"` or with `fullName` removed entirely — expect `{"success":false,"error":"..."}` describing what's wrong, and confirm **no row is added** to the sheet for a rejected request.

### D. Test the real booking modal
1. With `NEXT_PUBLIC_BOOKING_API_URL` set and the dev server restarted, open the site and click any "Book Discovery Call" button.
2. Complete Step 1 (date + time) and Step 2 (required fields), then on Step 3 click **Confirm Booking**.
3. Watch for the button switching to a spinner + "Booking...", then the success screen.
4. Check the **Leads** sheet for the new row.
5. To test the error path: temporarily set `NEXT_PUBLIC_BOOKING_API_URL` to an invalid URL, restart the dev server, and confirm clicking **Confirm Booking** shows the red error line instead of silently failing or crashing.

### E. Test on mobile / a second device
Since the endpoint is a public URL (not `localhost`), you can test from a phone on the same network once the site is deployed (or via a tunnel like `ngrok` for local dev) — the Apps Script side needs no changes for this.

## 5. Future-ready hooks

`Code.gs` has a single function, `runPostBookingHooks_(data)`, called right
after a booking is safely written to the sheet. It currently does nothing
— every call inside it is commented out — but the six features mentioned
in the brief (confirmation email, admin notification, Calendar event,
WhatsApp, CRM sync, analytics) each already have:

- A commented-out call in `runPostBookingHooks_`, wrapped in its own
  `try/catch` (so one failing hook can never cause a booking to be lost).
- A stub or example implementation below it — the email and Calendar ones
  are full working examples using Apps Script's built-in `GmailApp` /
  `CalendarApp` services; WhatsApp/CRM/analytics are placeholders (those
  need a specific external provider decided first).

Turning one on later is: implement/uncomment the function body, then
uncomment its one line in `runPostBookingHooks_`. No other changes to the
request handling, validation, or sheet-writing logic are needed.

## 6. Security notes

- The script validates required fields and email format **server-side**,
  not just in the browser — a request from anywhere else still has to
  pass the same checks.
- Field values are trimmed and capped at 2000 characters before being
  written to the sheet, so a malformed request can't create an
  oversized cell.
- "Anyone" access on the Web App deployment means anyone can *call the
  endpoint*; it does not expose the underlying spreadsheet or grant edit
  access to anyone but you. If you later want to restrict who can submit
  (e.g. only your own domain), that's an additional check you can add
  inside `doPost` — happy to add it as a follow-up if you want it now.
